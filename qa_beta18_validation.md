## Part 1 — Updated Requirements

### REQ-01: Streak Freeze Tokens
**Priority:** High | **Status:** Implemented + Verified

- FR-01.1: System SHALL award 1 streak freeze token when user reaches a 7-day streak milestone (7, 14, 21 …)
- FR-01.2: Freeze count SHALL be capped at 5 tokens maximum
- FR-01.3: If user misses exactly 1 day, the system SHALL auto-consume 1 freeze token and continue the streak
- FR-01.4: If user misses 2+ consecutive days, no freeze shall be applied and streak resets to 1
- FR-01.5: Sidebar SHALL display `🧊N` badge when user holds ≥1 freeze tokens
- FR-01.6: `User.streakFreezes Int @default(0)` column is the source of truth
- NFR-01.1: Freeze check executes within `updateUserProgress()` at session completion — zero extra DB calls

---

### REQ-02: Exam Countdown Widget
**Priority:** High | **Status:** Implemented + Verified

- FR-02.1: Dashboard SHALL show an Exam Countdown card with a date picker
- FR-02.2: Days remaining SHALL be calculated in the user's local timezone (local midnight basis)
- FR-02.3: Countdown SHALL display in indigo when ≥15 days remaining; red when <14 days
- FR-02.4: Sidebar SHALL show `N days until exam` badge derived from `User.examDate`
- FR-02.5: `PUT /api/user/exam-date` SHALL persist the exam date; `GET` returns it
- FR-02.6: Save SHALL use `router.refresh()` (not hard reload) to update server components
- FR-02.7: If save fails, an error message SHALL appear below the Save button
- FR-02.8: If exam date has passed, card SHALL display "Exam day has passed 🎉"
- NFR-02.1: Exam date is stored globally (one per user); multi-exam users set the nearest date

---

### REQ-03: Embedded Post-Wrong-Answer Knowledge Check
**Priority:** High | **Status:** Implemented + Verified

- FR-03.1: After an MCQ answer is submitted as **incorrect**, a comprehension check SHALL be auto-generated
- FR-03.2: The check SHALL consist of exactly 1 MCQ generated by `/api/ai/tutor/knowledge-check`
- FR-03.3: The check SHALL appear inline below the feedback card — no button click required
- FR-03.4: While the check is loading, a spinner with "Generating comprehension check…" SHALL appear
- FR-03.5: Student selects an answer; correct/incorrect highlight + explanation shows immediately
- FR-03.6: Check SHALL NOT appear after correct MCQ answers
- FR-03.7: Check SHALL NOT appear after FRQ/SAQ/LEQ/DBQ/CODING answers (open-ended only)
- FR-03.8: Fetch timeout SHALL be 25 seconds (AbortSignal.timeout)
- FR-03.9: On AI failure the check silently omits — user is not blocked from moving on
- NFR-03.1: Knowledge check is fire-and-forget; does not save to `TutorKnowledgeCheck` (practice context ≠ tutor context)

---

### REQ-04: Mastery Goal-Setting per Unit
**Priority:** High | **Status:** Implemented + Verified

- FR-04.1: Analytics page SHALL allow tapping a flag icon on any unit to open a goal-setting modal
- FR-04.2: Modal SHALL show current mastery, a slider (current+1 → 100), and an optional target date
- FR-04.3: Goal SHALL be stored in `MasteryGoal` table with `@@unique([userId, course, unit])`
- FR-04.4: A vertical marker SHALL appear on the unit's progress bar at the goal percentage
- FR-04.5: Below the bar, "N pts to goal" SHALL display if goal not yet reached
- FR-04.6: "Goal reached! 🎉" SHALL display when masteryScore ≥ targetScore
- FR-04.7: Goal deletion SHALL be confirmed by server before removing from UI
- FR-04.8: `GET /api/mastery-goal?course=` returns all goals for the course
- FR-04.9: `POST /api/mastery-goal` upserts by `(userId, course, unit)` — prevents cross-course overwrites
- FR-04.10: `DELETE /api/mastery-goal?unit=&course=` removes one goal; both params required
- NFR-04.1: Goal fetch runs in parallel with analytics fetch (single `Promise.all`)

---

### REQ-05: Weekly Leaderboard
**Priority:** High | **Status:** Implemented + Verified

- FR-05.1: Dashboard SHALL show a Weekly Leaderboard card for the currently selected course
- FR-05.2: Leaderboard SHALL rank users by XP earned this week (Monday 00:00 local server reset)
- FR-05.3: XP calculation: `correctAnswers × 10` per completed session
- FR-05.4: Top 10 entries SHALL display: rank emoji (🥇🥈🥉 for top 3), display name (first + last initial), XP badge
- FR-05.5: If current user is outside top 10, an additional row SHALL display their rank and XP
- FR-05.6: Display names SHALL be formatted as "FirstName L." to protect privacy
- FR-05.7: If API fetch fails, an error message SHALL display (not silent)
- FR-05.8: `lastName` null-safety: system SHALL use `lastName?.charAt(0) ?? "?"` to prevent crashes
- NFR-05.1: Leaderboard is anonymous-safe — no email or full name exposed

---

### REQ-06: Spaced Repetition Daily Review
**Priority:** High | **Status:** Implemented + Verified

- FR-06.1: Dashboard SHALL show a Daily Review card when ≥1 SRS-eligible questions exist
- FR-06.2: SRS window: MCQ questions answered **incorrectly** between 3 and 7 days ago (inclusive)
- FR-06.3: Review set SHALL be deduplicated by questionId (earliest wrong answer per question used)
- FR-06.4: Maximum 5 questions per Daily Review session
- FR-06.5: Review SHALL include `correctAnswer` and `explanation` fields (returned by `/api/review`)
- FR-06.6: Selecting the correct answer SHALL highlight it green and show the explanation
- FR-06.7: Selecting the wrong answer SHALL highlight it red, correct option green, then show explanation
- FR-06.8: Score SHALL be tracked per session and displayed at the end
- FR-06.9: `+N XP Daily Boost` badge SHALL appear on completion (5 XP per correct answer)
- FR-06.10: If API fetch fails, error message SHALL appear (not silent spinner)

---

### REQ-07: Socratic 3-Level Hints
**Priority:** Medium | **Status:** Implemented + Verified

- FR-07.1: `POST /api/ai/hint` SHALL accept an optional `hintLevel` param (1, 2, or 3)
- FR-07.2: Level 1 — Concept reminder: identifies the AP topic tested; does NOT hint at answer
- FR-07.3: Level 2 — Process clue: explains how to approach the question (strategy, not answer)
- FR-07.4: Level 3 — Scaffold: step-by-step reasoning ending in a guiding question; does NOT reveal the letter
- FR-07.5: `hintLevel` is clamped to [1, 3] server-side; defaults to 1 if absent
- FR-07.6: Response SHALL include `{ hint: string, hintLevel: number }`
- NFR-07.1: Socratic prompts use `callAIWithCascade` — same provider cascade as other AI features

---

### REQ-08: Voice Input to Sage
**Priority:** Medium | **Status:** Implemented + Verified

- FR-08.1: AI Tutor page SHALL show a microphone button beside the Send button
- FR-08.2: Clicking Mic button SHALL request browser speech recognition (Web Speech API)
- FR-08.3: If browser does not support speech recognition, a toast SHALL appear: "Voice not supported"
- FR-08.4: If microphone permission is denied, a toast SHALL appear: "Microphone denied — allow in settings"
- FR-08.5: While recording, button SHALL show MicOff icon with `variant="destructive"` styling
- FR-08.6: Transcript SHALL be appended to current input (space-separated if input has content)
- FR-08.7: After recognition ends (natural or error), recording state SHALL reset to false
- FR-08.8: Voice button SHALL be disabled when daily limit is reached
- NFR-08.1: No server changes required — purely client-side Web Speech API
- NFR-08.2: Transcript is validated via optional chaining before appending

## Part 4 — QA Audit Report

### Defects Found and Fixed (Beta 1.8 cycle)

| D-ID | Severity | Component | Description | Root Cause | Fix |
|------|----------|-----------|-------------|------------|-----|
| D01 | 🔴 Critical | `daily-review-card.tsx` | Feature entirely non-functional: `correctAnswer` missing from API response, `handleAnswer()` dead code, score never tracked | `/api/review` did not select `correctAnswer` or `explanation`; UI called `next()` not `handleAnswer()` | Added `correctAnswer`+`explanation` to review API select; full component rewrite with proper `handleAnswer` → score tracking |
| D02 | 🔴 Critical | `schema.prisma` | `MasteryGoal` unique constraint was `(userId, unit)` — same unit key exists across courses (e.g. UNIT_1 in AP World and SAT), causing cross-course goal overwrites | Design oversight — unit enum values overlap across courses | Changed to `@@unique([userId, course, unit])`; updated upsert key and DELETE query |
| D03 | 🔴 Critical | `api/leaderboard/route.ts:54` | `lastName.charAt(0)` crashes if user has null/empty `lastName` (possible for Google OAuth users) | Google OAuth users can have missing lastName | Changed to `lastName?.charAt(0) ?? "?"` |
| D04 | 🟠 High | `exam-countdown-setter.tsx:31` | Countdown calculated using UTC date math — off-by-1 day for users in eastern/western timezones | Used `Date.now()` vs `new Date(dateString)` without aligning timezone | Rewrote to parse as local midnight: `new Date(y, m-1, d)` vs `today.setHours(0,0,0,0)` |
| D05 | 🟠 High | `exam-countdown-setter.tsx:44` | `window.location.reload()` on save caused hard page reload (jarring UX, breaks history) | Used browser reload instead of Next.js router | Replaced with `router.refresh()` which re-fetches server components only |
