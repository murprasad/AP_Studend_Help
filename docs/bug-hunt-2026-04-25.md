## Value-prioritization rubric

For every defect, score by:
1. **Severity** — P0 (breaks paying user / data loss) > P1 (breaks core flow) > P2 (degraded UX) > P3 (polish)
2. **Frequency** — every user / every session > some users / some sessions > rare conditions
3. **Conversion impact** — touches sign-up/checkout/practice (the funnel) > touches admin/internal
4. **Reversibility** — silent data loss > visible error > UI awkwardness

A P3 polish issue on a rarely-visited page = bottom of queue.
A P2 issue on /pricing seen by every prospect = mid-queue, ship soon.
A P0 cold-start 5xx affecting 10% of post-deploy sessions = top of queue, ship now.

## A1 Findings — categorical totals (~660 raw findings from 10 parallel agents)

### Cat 1 — Accessibility (35 findings)

**Real:** Touch targets <48px (Sage chat buttons, AP-season banner dismiss), modals without focus trap (first-answer reward, auto-launch nudge), icon-only buttons without aria-label, color-only state indicators (billing toggle), missing form/label associations (analytics goal modal), generic image alt text on practice page stimuli.

**Top-3 highest-value:**
- AP-season banner dismiss button: ~16-20px target on mobile (banner shown to every visitor)
- First-answer reward modal: no focus trap or restore-focus (every new user)
- Sage chat buttons: 36px instead of 48px (every premium user)

### Cat 2 — Security (~75 findings)

**Real:**
- Open redirect risk in `/api/checkout` and `/api/billing/portal` (origin header not validated)
- IDOR risk: `/api/mastery-goal/[id]` and `/api/frq/[id]` — ownership not always verified
- 78 POST routes without input validation (no zod/schema, no Content-Length cap)
- Timing-attack vector on `CRON_SECRET` comparisons (`!==` not constant-time)
- SQL injection via `$executeRawUnsafe` in `/api/diagnostic/route.ts:126` and `/api/questions/[id]/report/route.ts:45`
- Mass assignment in `/api/community/threads/route.ts:40` (Prisma `.create` accepts raw body)
- `/api/admin/reset-test-users/route.ts` accessible in production with hardcoded test emails
- Stripe webhook orphan-account vector via email fallback (already known — saravanab2v case)

**Top-3 highest-value:**
- SQL injection in `/api/diagnostic` and `/api/questions/[id]/report` (P0)
- Open redirect in `/api/checkout` (P1, attacker can redirect users to phishing site)
- Missing input validation on 78 POST routes (P1, batch fix needed)

### Cat 3 — Error handling / fail-silent (~120 findings)

**Real:**
- 16 `.catch(() => {})` empty catches swallowing errors silently
- 12 `console.error/warn` in API routes instead of Sentry
- 8 unsafe `JSON.parse` without try/catch
- 7 feature flags checked AFTER expensive operations
- 6 useEffect fetches rendering null on error (invisible UX)
- 5 Promise.allSettled returning partial data without user feedback
- Dashboard 5xx (FIXED in Beta 7.2)

**Top-3 highest-value:**
- Webhook handler `console.warn` on critical failures (P1, Stripe state drift goes silent)
- Practice-page submit error generic "Check your connection" for any failure (P1, hides real cause)
- Sage chat provider failures via console.warn only (P2, no alerting on cascading provider failures)

### Cat 4 — Performance (~80 findings)

**Real:**
- 30+ `findMany()` without `.select()` returning full rows (heavy payloads)
- 6 N+1 query loops in cron routes (registration-stall, daily-quiz, onboarding-bounce, weekly-progress-digest)
- 5 missing compound indexes (User, PracticeSession, StudentResponse, TutorConversation, DiagnosticResult)
- 4 unbounded `findMany()` calls (could load entire question table)
- localStorage reads in render path (sage-chat, exit-intent-capture, goal-card)
- No Cache-Control headers on safe-to-cache GETs

**Top-3 highest-value:**
- `practice/route.ts:98` findMany returns full Question rows including stimulusImageUrl + options (every practice session)
- N+1 in registration-stall pass 2 (500 DB roundtrips on every cron run)
- Missing index on PracticeSession(userId,status,completedAt) (used in every dashboard load)

### Cat 5 — Stripe + billing (~50 findings)

**Real:**

- No event_id idempotency (duplicate webhook delivery replays upserts)
- Cancel endpoint flips DB before verifying Stripe state
- Period-end calculations inconsistent on cancel/proration paths

**Top-3 highest-value:**
- Webhook returning 500 → Stripe stops retrying (P0, real revenue impact — failed renewals never recover)
- No `invoice.payment_failed` handling (P0, downgrades never happen on real card failures)
- No event_id dedup (P1, double-processing risk)

### Cat 6 — AI provider robustness (~70 findings)

**Real:**

**Top-3 highest-value:**
- Streaming endpoint without timeout — connections can hang indefinitely (P0, hangs Sage chat)
- AI cascade `break` on hard fail prevents trying next provider (P1, single provider death = no AI)
- Question generation saves before validation = bad questions in DB if validator rejects (P1, content quality)
